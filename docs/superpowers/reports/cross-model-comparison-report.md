**代码生成 +3**: unsloth MTP 在代码生成上略好，可能与 MTP（多token预测）训练有关。

**代码补全 -5**: reasoning-distilled 略优，补全任务对推理能力要求较高。

## 3. Agent能力对比 (61.5 vs 53.9) 🚀

| 子类别 | reasoning-distilled | unsloth MTP | 差距 |
|--------|:---:|:---:|:----:|
| Function Calling | 38/60 | 38/60 | 0 |
| 工具选择 | 35/45 | 35/45 | 0 |
| 多步推理 | 19/40 | **22/40** | **+3** |
| **指令遵循** | **31/45** | **45/45** | **+14 🚀** |
| 文件系统操作 | 12/45 | 12/45 | 0 |
| Shell命令执行 | 36/40 | 36/40 | 0 |
| 工具编排 | 33/80 | **37/80** | **+4** |
| **多轮对话** | **9/35** | **11/35** | **+2** |
| **结构化输出** | **9/25** | **25/25** | **+16 🚀** |
| 长任务规划 | 14/30 | 14/30 | 0 |

### 关键发现

**结构化输出 25/25 (100%)**: 这是反转最大的子类别。reasoning-distilled 模型由于 thinking block 干扰（content 为空的响应），JSON 提取困难。而 unsloth MTP 可以干净地返回正确 JSON。

**指令遵循 45/45 (100%)**: reasoning-distilled 模型的 thinking block 经常导致输出不符合格式要求，而非 reasoning 模型则能严格遵循指令。

**多轮对话 11/35 vs 9/35**: unsloth MTP 稍好，但多轮对话仍然是两模型的薄弱环节。

P5(多轮对话) 改进有限的原因是: 即使是标准模型，多轮 tool_call_id 和参数匹配仍然困难。C-1/C-2 修复提供了基础支持（tool_call_id 传递 + 文本降级），但进一步提高需要更高级的语义匹配。
